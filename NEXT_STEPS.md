## 1. Защитить `kotc_sessions`

Статус: реализовано в текущей ветке — 2026-03-15.

### Почему

Сейчас realtime-синхронизация счёта остаётся самым слабым местом: знание `room_code` даёт фактический доступ к состоянию комнаты. Для локального клуба это терпимо, для публичного использования нет.

### Что менять

- Убрать инструкцию с `ALTER TABLE kotc_sessions DISABLE ROW LEVEL SECURITY`.
- Перестроить слой комнаты вокруг явного владения:
  - `create_room`
  - `get_room_state`
  - `push_room_state`
  - `rotate_room_secret`
- Если нужен именно безопасный режим без полноценного логина:
  - хранить `owner_secret_hash` и `viewer_secret_hash` на сервере;
  - отдавать секрет только в момент создания комнаты;
  - использовать `SECURITY DEFINER` RPC вместо прямой записи в таблицу.
- Если нужен realtime без открытой таблицы:
  - либо перейти с `postgres_changes` на `broadcast`;
  - либо сделать polling через RPC раз в 1-2 секунды для счёта и таймеров.

### Где менять

- `assets/js/integrations.js`:
  - RPC-подключение комнат
  - UI-карточку Supabase
- `assets/js/screens/roster.js`:
  - размещение и поведение административных блоков
- `supabase_migration.sql`:
  - схему `kotc_sessions`
  - новые RPC
  - RLS-политики

### Критерий готовности

- Нельзя читать и писать состояние комнаты только по `room_code`.
- SQL-инструкция в UI больше не предлагает отключать RLS.
- Комната создаётся и подключается через явный секрет или токен владельца.

## 2. Разрезать монолит `index.html`

Статус: реализовано в текущей ветке — 2026-03-15.

### Почему

Главная техническая цена проекта сейчас — слишком большой single-file frontend. Любая правка опасна, потому что логика регистрации, таймеров, ростера, PWA и sync-подсистем живёт в одном месте.

### Что менять

Минимальная декомпозиция без React/Vue:

- `assets/app.css`
- `assets/js/main.js`
- `assets/js/core.js`
- `assets/js/registration.js`
- `assets/js/screens/*.js`
- `assets/js/integrations.js`
- `assets/js/domain/timers.js`
- `assets/js/ui/roster-auth.js`
- `assets/js/runtime.js`

### Правило переноса

- Не переписывать всё заново.
- Переносить код блоками "как есть", затем чистить.
- Сначала вынести только JS и CSS, HTML-структуру оставить в `index.html`.

### Критерий готовности

- В `index.html` остаются только разметка, подключения ресурсов и минимальный bootstrap.
- Основные подсистемы вынесены из inline-скрипта в отдельные файлы.
- Поведение приложения сохраняется без перехода на новый фреймворк или сборщик.

Дополнительно реализовано:

- `timer` вынесен в `assets/js/domain/timers.js`;
- локальная защита ростера вынесена в `assets/js/ui/roster-auth.js`;
- `screens.js` разрезан на `assets/js/screens/*.js`;
- запуск переведён на `assets/js/main.js` + `type="module"` без внедрения полноценного фреймворка.

## 3. Сделать админский workflow

Статус: реализовано в текущей ветке — 2026-03-15.

### Почему

SQL уже умеет больше, чем UI. Есть `approve_player_request`, `merge_players`, `safe_cancel_registration`, `merge_audit`, но в интерфейсе нет полноценного контура модерации.

### Что менять

- Добавить в ростер или отдельную админ-вкладку:
  - список `pending` заявок;
  - одобрение заявки;
  - отклонение заявки;

  - просмотр `merge_audit`.
- Для временных игроков:
  - пометка в UI;
  - быстрый merge в реального игрока;
  - предупреждение при попытке завершить турнир с большим числом temporary-профилей.

### Где менять

- `assets/js/screens/roster.js`
- `assets/js/registration.js`
- `assets/js/domain/players.js`, если понадобится нормализовать операции с temporary-профилями

### Критерий готовности

- Заявка не требует ручной работы через SQL Editor.
- Temporary-профили можно безопасно склеивать из интерфейса.

## 4. Добавить browser smoke-тесты

Статус: реализовано в текущей ветке — 2026-03-15 (Playwright, CI job, npm-скрипты).

### Почему

Сейчас есть только статическая валидация файлов. Она ловит синтаксис и отсутствие артефактов, но не ловит поломки интерфейса.

Ручной headless smoke-прогон уже был выполнен 15 марта 2026 года и не выявил регрессий приложения:

- приложение загрузилось без bootstrap-ошибки;
- вкладки `home`, `players`, `svod`, `stats`, `roster` открылись корректно;
- `service worker` зарегистрировался и стал active;
- не было `console error`, `pageerror`, `requestfailed` и `4xx/5xx` ответов по ресурсам приложения.

По итогам этого прогона стало понятно, что основные оставшиеся правки здесь не продуктовые, а инфраструктурные.

### Что менять

Добавить Playwright или аналогичный lightweight e2e слой в сам репозиторий.

Нужные доработки по итогам smoke-прогона:

- зафиксировать smoke-тест в репозитории, а не гонять ad hoc-скриптом;
- добавить `package.json`/npm-script для запуска browser smoke локально;
- добавить CI job для smoke-теста отдельно от статической валидации;
- задокументировать browser dependencies для Chromium;
- не полагаться на временный локальный обход через `LD_LIBRARY_PATH`;
- исключить временные артефакты smoke-прогонов из рабочего дерева и коммитов:
  - `.tmp-smoke/`
  - `test-results/`
  - ad hoc-файлы из временных директорий

Минимальный сценарий:

1. Открыть приложение.
2. Установить пароль ростера.
3. Создать турнир локально.
4. Добавить игрока.
5. Открыть регистрацию.
6. Сохранить турнир.
7. Проверить загрузку `manifest` и регистрацию service worker.

Минимум, который уже стоит покрыть дополнительно к текущему ручному прогону:

1. Проверка module bootstrap через `assets/js/main.js`.
2. Проверка открытия ростера с локальной парольной защитой.
3. Проверка локального создания турнира из home/roster flow.
4. Проверка регистрации/быстрой записи игрока.
5. Проверка работы после перезагрузки страницы с уже зарегистрированным `service worker`.

### Необходимая информация

Чтобы smoke-тест был воспроизводимым, в проекте нужно явно зафиксировать:

- какой runner используется: `Playwright`;
- как ставится браузер: `npx playwright install chromium`;
- какие системные зависимости нужны для Chromium в Linux-окружении;
- где хранить и как очищать test artifacts;
- какой локальный URL считается тестовым, например `http://127.0.0.1:9011/`;
- какие внешние запросы допустимо игнорировать в smoke, например шрифты и Google auth SDK.

На машине, где делался текущий прогон, не хватало системных библиотек Chromium:

- `libnspr4`
- `libnss3`
- `libsmime3`
- `libasound.so.2`

Это нужно учесть либо в документации, либо сразу в CI/container image.

### Критерий готовности

- Smoke-тест живёт в репозитории и запускается одной командой.
- CI валится при поломке базовой навигации или ключевых пользовательских потоков.
- Для запуска теста не нужен ручной локальный обход с временными библиотеками и ad hoc-скриптами.

## 5. Вынести конфиг окружения

Статус: реализовано в текущей ветке — 2026-03-15 (config.js / config.example.js, window.APP_CONFIG).

### Почему

Сейчас URL проекта Supabase и `anonKey` жёстко записаны в `assets/js/integrations/config.js`. Для статического сайта это допустимо, но плохо для поддержки нескольких окружений.

### Что менять

- Создать `config.example.js` и `config.js`.
- Либо использовать `window.APP_CONFIG`.
- Все runtime-настройки читать из одного места.

Пример целевой структуры:

```js
window.APP_CONFIG = {
  supabaseUrl: '',
  supabaseAnonKey: '',
  googleClientId: ''
};
```

### Критерий готовности

- Для смены окружения не нужно редактировать основной файл приложения.

## 6. Нормализовать фронтенд-данные

Статус: реализовано в текущей ветке — 2026-03-18.

### Почему

Во фронтенде есть историческая смесь полей:

- `tournaments` и `tournaments_played`
- `totalPts` и `total_pts`
- локальные игроки и Supabase-игроки имеют разные формы объектов

Это уже частично закрыто адаптерами, но лучше дожать до одного формата.

### Что менять

- Ввести один canonical shape для игрока.
- Сделать адаптеры:
  - `fromSupabasePlayer`
  - `fromLocalPlayer`
  - `toLocalPlayer`
- Прекратить разносить совместимость по всему коду.

### Критерий готовности

- Внутри приложения все игроки имеют один формат объекта.
- Совместимость со старым `localStorage` сидит только в адаптере миграции.

Дополнительно реализовано:

- `assets/js/domain/players.js` содержит canonical shape (задокументирован в шапке файла) и три адаптера: `fromLocalPlayer`, `toLocalPlayer`, `fromSupabasePlayer`.
- Поля `tournaments_played` / `total_pts` нормализуются только внутри адаптеров; остальной код работает исключительно с `tournaments` / `totalPts`.
